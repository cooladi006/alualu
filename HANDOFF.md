## 1. What this is

**Aditya Jain** (Chartered Accountant) is setting up an **Alu-Alu pharmaceutical blister
packaging plant in Chennai**. "Alu-Alu Command" is a **first-principles financial
modelling & decision-support tool** — a zero-dependency, single-file PWA — that lets him
feed real vendor rates and actuals, play with inputs, and immediately see whether he is on
target for his return goals and **which lever to pull** when he is off.

It is a control tool, not a reporting dashboard. The north star: *enter data → see breakeven,
ROI, and the limits to hold → set targets → later, tweak targets against actuals.*

There is a **separate, unrelated app called `fitaditya`** in a different repo. Never confuse
the two. This project's repo is **`alualu`**.

### Machine-evaluation side-thread (context only)
The conversation opened with a question about whether **Koley Machinery (₹90 L)** can produce
reliable Alu-Alu / blister foil versus **Alpha Roto (₹2 Cr)** and **Sudha (₹2.75 Cr)**. The
three machine technical-spec PDFs live in the project files. That hardware due-diligence is a
separate analysis; the app work below is what we actually built. (If you want to resume the

## 3. The financial model (first principles)

All money is held in **rupees** internally; `cr(x)` formats to crore (÷1e7, 2 dp), `inr(x)`
formats Indian-grouped rupees, `dec(x,n)` fixed decimals, `pct(x)` percent, `nf(x)` parses a
numeric field.

### Raw material cost build-up (REV 11 economics)
- `GSM = thickness(µm) × density(g/cm³)`; `cost/m² = (GSM/1000) × rate`.
- Substrates (defaults): **OPA** (25µm, 1.15, ₹260/kg), **Aluminium foil 8021** (50µm, 2.71,
  ₹560/kg — **72.49% of RM cost**), **PVC** (60µm, 1.35, ₹220/kg).
- **Adhesive** blend ratio 100:15:100 (KUB130 ₹420 : Hardener/NeoForce ₹420 : Ethyl Acetate ₹115),
  ~50 kg mix/tonne → blended ≈ **₹278/kg**, ~6.74 dry GSM, adhesive ≈ ₹13.9/kg.
- `yield ≈ 0.252 kg/m²`; `cost/kg = cost/m² ÷ yield`.
- **Ideal RM ≈ ₹415.41/kg**, then **finalRM = ideal × (1 + wastage%)**; at 6% wastage → **₹440.34/kg
  inclusive of wastage** (this is the number that drives the whole report).
- Selling price default **₹535/kg**.

### P&L (per volume column)
`salesYr`, `rmYr`, `fixedYr`, `varYr`, `ebit` (operating profit), `patEq = ebit×(1−tax)`.

### Working capital — GST-blocked across the whole cycle (REV 11)
GST (`gst`, default 18%) is paid on foil/RM at purchase but recovered only when customers
pay, so it sits in **every** WC stage. All stages are GST-inclusive:
- `rmInv  = annualRM   × (1+gst) × rmStockDays/365` (30)
- `fgInv  = annualCOGS × (1+gst) × fgDays/365` (15); `annualCOGS = salesYr − ebit`
- `recv   = annualSales × (1+gst) × debtorDays/365` (90)
- `pay    = (nonFoilRM × otherCreditorDays + foilRM × foilCreditDays) × (1+gst) / 365`
  where `foilShareOfRM = 0.7249`; **foil is bought on advance (`foilCreditDays = 0`)** so it
  earns no supplier credit; non-foil RM gets `otherCreditorDays` (10).
- `netWC = rmInv + fgInv + recv − pay` (≈ **₹13.09 Cr at 50T** with GST).
- **Cash conversion cycle = rmStockDays + fgDays + debtorDays − otherCreditorDays = 125 days.**

### Capital & returns
- **Fixed assets = ₹3.45 Cr** (11 line items: machine 1.25, slitting 0.30, HVAC 0.25, QC lab 0.20,
  epoxy 0.10, adhesive mix 0.05, cabins/electricals/furniture 0.50, doctoring 0.10, genset 0.10,
  roll stand 0.10, misc 0.50).
- **Total Capital Employed `I` = fixedAssets + netWC**.
- **RoCE `roiCap` = patEq / I**.
- Leverage with an **editable equity cap** (`S.equityCap`, default 8 Cr):
  `equityUsed = min(equityCap, I)`, `debtCap = max(0, I − equityCap)`,
  **RoE `roeCap` = (ebit − debtCap×interest)×(1−tax) / equityUsed**, `deCap = debtCap/equityUsed`.
- Defaults: target ROI 30%, interest 11%, equityCap 8 Cr, maxDE 2, tax 0%.

### Headline results at defaults (volumes 25 / 50 / 75 / 100 T-month)
- Total Capital Employed: **7.05 / 10.58 / 14.12 / 17.66 Cr**
- RoCE: **10% / 27% / 35% / 39%** (crosses 30% ≈ 60.5T)
- RoE (8 Cr equity + debt): **10% / 32% / 53% / 74%**
- Self-funded ceiling on 8 Cr (no debt, interpolated): **≈ 31.7 T/mo, ≈ ₹18.66 Cr turnover, RoCE 16%**

### The control / target maths (newest, the current focus)
Module-level functions, used by both the live app and the PDF:

- **`breakeven(M)`** → contribution/kg = price − finalRM − varPerKg; breakeven qty = fixedMo /
  contribution; breakeven sales = qty × price; margin of safety vs focus volume.
  At 50T: contribution **₹80/kg**, breakeven **20.7 T/mo ≈ ₹1.01 Cr/mo**, safety **+59%**.
- **`controlLimits(M)`** → the single-lever ceilings to hit the target RoCE, *holding the others*:
  - `maxFixedMo` = (salesYr − rmYr − varYr − tgt×I/(1−tax)) / 12  → **≤ ₹13.59 L/mo** (at 16.55 → off)
  - `maxDeb` (debtor days): from `I_target = PAT/tgt`, back out recv → **≤ 60 days** (at 75 → off)
  - `maxInv` (RM+FG days, scaled together) → **≤ 28 days** (at 45 → off)
  - `minVol` (interpolate roiCap across tiers to where it crosses target) → **≥ 60.5 T** (at 50 → off)
  - Each returns an `...Ok` boolean for green/red status.
- **`selfFundCeiling(M)`** → interpolates the tonnage where `I = equityCap` (where own money runs
  out), with the P&L there.
- **`growthNoDebt(M)`** → bootstrap simulation (kept in code but **no longer surfaced** — it was
  replaced because "grow to X in N years" confused the user; see history).

## 4. App architecture

Single `<script>` in `index.html`. Pure vanilla — **no libraries, no build step, no network**.

- **State `S`** (persisted to `localStorage['alualu_state_v1']`): `substrates[3]` (each
  `{name,thickness,density,sel,vendors[{name,rate}]}`), `adh{...}`, `wastage`, `price`,
  `cols[4]` (each `{vol, fixed[14 line items], variable[3]}` — **volumes are editable**),
  `focus` (0–3), `assets[11]`, `rmDays/fgDays/debDays/credDays`, `targetROI`, `tax`, `interest`,
  `equityCap`, `maxDE`, `revision`, `tab`, `screen`, `rep`.
- **`compute()`** returns `{base, tiers[4], sel = tiers[focus], threshold, Ecap, ...}`. Every tier
  carries salesYr/rmYr/fixedYr/varYr/ebit/patEq/I/roiCap/equityUsed/debtCap/deCap/roeCap/wc/
  rmInv/fgInv/recv/pay/conv, etc.
- **`render()`** rebuilds the DOM from `S`; **`save()`** writes localStorage; **`load()`** reads it
  at startup. Autosave on every render. "Reset all data" link on the Targets screen.
- **Vendor system:** tap a circle to select which vendor's rate builds the cost sheet; +Add /
  delete vendor; selected row tinted green.
- **Three tabs** (segmented control): **ROI** (dashboard matrix), **Enter data** (screens:
  price / materials / expenses / capital / wc / funding / targets), **Report** (segmented:
  Viability / Cost sheet).
- The **equity cap is editable** on *Enter data → Funding & leverage* (`field("equityCap", …)`);
  the entire report recomputes from it (verified: 9 Cr → ceiling 38.8T; <7.05 Cr → "debt from
  the start").
- **Frosted bottom action bar** on the Report tab: full-width green "Share as PDF" + grey "Print".
  `sharePDF()` builds a `File` and calls `navigator.share({files})` directly (no canShare gate) so
  WhatsApp/Mail appears on phones; falls back to download on desktop/incognito/in-app browsers.

### The PDF generators (hand-rolled, zero-dependency)
A4 595×842 pt, Helvetica / Helvetica-Bold, **WinAnsi, ASCII-sanitised** (`san()` strips/normalises
non-ASCII; there is **no** unicode font). A page factory `P()` exposes `rect/box/line/txt/rt/ctr/
wrap`; outer pure helpers `hx/san/escp/tw`. `tileRow()` draws the KPI tile rows.

- **`buildPDF(M)` = 4-page viability report** (current):
  1. **Overview** — two key-assumption boxes (KEY ASSUMPTIONS + RAW MATERIALS with the Ideal →
     +wastage% → RM-incl-wastage working), the returns-by-volume matrix (P&L / investment / funding,
     focus column tinted, % vs sales in grey), and four "WHAT IT MEANS" analysis boxes.
  2. **Targets, breakeven & control** — `drawPageCtrl`: "where you stand" tiles, the BREAK-EVEN
     panel (volume + sales + contribution + fixed + safety), the CONTROL LIMITS table ("to hold
     30% at 50T — fix any ONE": volume / fixed / debtor days / inventory days, each with limit,
     current, and on-track/off status), and a plain-English VERDICT.
  3. **Capital & self-funding** — `drawPage2`: fixed-asset build-up box + working-capital workings
     box (the "Annual RM /365 × days" formula style), the self-funded-ceiling band, and the
     "WILL YOU NEED OUTSIDE MONEY?" table (capital / equity / outside need / profit/yr / repaid-in
     months, driven entirely by the cycle days).
  4. **Monthly cost break-up** — `drawPage3`: every fixed & variable line across all four volumes,
     plus conversion / RM / selling price per kg, and a "READING IT" note.
- **`buildCostPDF(M)` = 1-page cost sheet** — replicates the user's "ALU-ALU RAW MATERIAL COST
  REPORT / EXECUTIVE SUMMARY · REVISION 10" (6 KPI boxes + two black-bar sections).

PDF object layout for the 4-page report: catalog(1), pages(2, Kids [3,7,9,11] Count 4),
page+stream pairs 3/4, 7/8, 9/10, 11/12, fonts 5 & 6. (The cost PDF is a separate single-page
assembly with `Kids [3 0 R] Count 1` — don't confuse the two when editing.)

### Design system (Apple/Tesla/Microsoft-CEO tier)
Pure white bg, SF Pro system font in-app. Tokens used in PDFs: GREEN `#0E7C57`, RED `#C5304A`,
INK `#1D1D1F`, INK2 `#6E6E73`, INK3 `#9A9AA0`, HAIR `#D6D6DA`, HAIRS `#E8E8EC`, BAND `#F2F2F4`,
SELC `#EDF6F1` (focus tint), GT `#EEF6F1` (green tint), WHITE, GREY `#8E8E93`. Tabular numerals,
single green accent, minimal formatting.

## 5. Build & test workflow (reproduce in Claude Code)

The app is one file, so the loop is simple. The PDF/JS validation harness used here:

1. **Edit** `index.html` (the `<script>` block).
2. **Extract the JS** and syntax-check before anything else:
   ```bash
   python3 -c "import re;open('_v.js','w').write(re.search(r'<script>\n(.*)\n</script>', open('index.html').read(), re.S).group(1))"
   node --check _v.js
   ```
3. **Headless harness** — stub the browser, then exercise every screen and generate the PDFs:
   ```js
   // stub document/navigator/window/localStorage/Blob/File/URL, then:  cat _v.js  then:
   load();
   ["dashboard","inputs","report"].forEach(x=>{S.tab=x;S.screen=null;render();});
   ["price","materials","expenses","capital","wc","funding","targets"].forEach(sc=>{S.tab="inputs";S.screen=sc;render();});
   S.tab="report";["viability","cost"].forEach(r=>{S.rep=r;render();});
   require("fs").writeFileSync("/tmp/v.pdf", Buffer.from(buildPDF(compute())._b));
   ```
4. **Verify pages & render to PNG** to eyeball layout:
   ```bash
   python3 -c "import pypdf;print(len(pypdf.PdfReader('/tmp/v.pdf').pages))"   # expect 4
   pdftoppm -png -r 150 /tmp/v.pdf /tmp/v                                      # then view the PNGs
   ```
   `pdftoppm` is the reliable renderer. `wkhtmltoimage` exists but mangles CSS grid / backdrop-filter
   — don't trust it for the in-app screens. The real device renders correctly.

### Editing tips that saved pain
- When patching the JS with Python, the PDF assembly strings contain **literal `\n`** (backslash-n).
  Use **raw strings** (`r'''...'''`) so Python doesn't turn them into newlines, and match the
  **exact** code (e.g. the page-creation is on **one line**: `var p1=P();drawPage1(p1);var p2=…`).
- After a `str_replace`, earlier line numbers are stale — re-view before the next edit.
- `node --check` reporting OK only proves syntax; always render to confirm layout/fit.
- Keep the viability report to its page count — verify `pages == 4`, not 5.
